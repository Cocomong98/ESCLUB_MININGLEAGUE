# CSP 강화 롤아웃 계획서

작성일: 2026-03-13 (KST)

## 1) 배경

현재 `app.py`의 전역 CSP는 운영 호환성을 위해 완화되어 있음:

- `script-src`에 `'unsafe-inline'`, `'unsafe-eval'` 포함
- admin 페이지가 외부 CDN(Tailwind/Alpine)에 의존

이 상태는 기능 호환성은 높지만 XSS/공급망 리스크 방어력이 낮다.

## 2) 변경하지 않을 때의 위협

1. XSS 1회 성공 시 브라우저에서 악성 스크립트 실행 가능성이 높음
2. 외부 CDN 변조(공급망) 시 악성 코드가 실행될 수 있음
3. 관리자 세션 문맥에서 관리자 API 오남용 위험이 커짐
4. 내부 NAS 환경에서는 확률이 낮을 수 있으나, 인터넷 노출 시 영향도는 중~높음

## 3) 목표

1. 일반 사용자 페이지는 강한 CSP 적용
2. 관리자 페이지는 별도 정책으로 분리 운영
3. 기능 중단 없이 단계적으로 강화

## 4) 권장 적용 순서 (A안)

### 단계 1: CSP 경로 분리 (즉시)

대상:
- 일반 페이지: `/tables`, `/dashboard/*`, `/hall-of-fame`, `/data/*`
- 관리자: `/admin`, `/admin-panel.js`, `/api/login` 등 admin 연계 경로

작업:
- 일반/관리자 CSP를 분리
- 기존 전역 단일 정책을 라우트별 정책으로 교체

운영 영향:
- 낮음 (동작은 대부분 기존과 동일)

### 단계 2: 일반 페이지 CSP 강화 (선행 적용)

작업:
- 일반 페이지 `script-src`에서 `'unsafe-eval'` 제거
- 필요 시 `'unsafe-inline'`도 제거 (사전 검증 후)
- 외부 도메인 화이트리스트 최소화

운영 영향:
- 중간 (일부 페이지 스크립트 차단 가능성)

### 단계 3: Report-Only 검증 (1~2일)

작업:
- `Content-Security-Policy-Report-Only`로 먼저 적용
- 위반 로그 수집 후 누락 도메인/정책 보정

운영 영향:
- 매우 낮음 (차단 없이 모니터링)

### 단계 4: 관리자 페이지 의존성 제거 후 강화

작업:
- admin의 CDN 자산(Tailwind/Alpine) 로컬 정적 자산으로 전환
- admin CSP에서도 `unsafe-*` 축소

운영 영향:
- 중간 (admin 화면/입력 기능 회귀 가능)

## 5) 일정 가이드

### 빠른 적용 (1일)

1. 오전: 단계 1 + 단계 3(Report-Only)
2. 오후: 로그 확인 후 일반 페이지 강제 전환

### 안정 적용 (2~3일)

1. 1일차: 단계 1 + Report-Only
2. 2일차: 일반 페이지 강제 전환
3. 3일차: admin CDN 로컬화 + admin 강화

## 6) 변경 시 발생 가능한 서비스 지장

1. 일반 페이지 JS/CSS 차단으로 렌더링 실패 가능
2. admin 로그인/입력/저장 기능 중단 가능
3. GTM/통계 등 외부 스크립트 누락 가능
4. 백엔드 크롤링/스케줄러에는 영향 없음 (CSP는 브라우저 정책)

## 7) 검증 체크리스트

1. 일반 페이지:
- `/tables`
- `/dashboard/<id>`
- `/dashboard/<id>/squad`
- `/hall-of-fame`

2. 관리자:
- `/admin` 접속/로그인
- 구단주 수정/저장
- 시즌 생성/수정

3. 브라우저 콘솔:
- CSP 위반 로그 존재 여부
- 차단된 리소스 출처 확인

## 8) 롤백 기준

1. 핵심 페이지 1개 이상 미렌더링 시 즉시 이전 CSP 정책으로 롤백
2. admin 저장 기능 실패 시 admin 정책만 우선 완화
3. 롤백은 코드 전체가 아닌 CSP 헤더 정책만 최소 단위로 수행

## 9) 구현 메모

- 현행 CSP 위치: `app.py` `@app.after_request`
- 적용 원칙:
  - “기본은 차단, 필요한 것만 허용”
  - “일반 페이지를 먼저 강화, admin은 분리 후 순차 강화”
